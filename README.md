## What is SignWatch?

SignWatch is a crowdsourced road sign monitoring PWA. You drive around with it open and it automatically photographs nearby signs, sends them to Claude AI for analysis, and flags any that are obscured or missing on a map. Flagged signs can be reported to the city with one tap.

## What Was Built

- **React PWA** (Vite) installable on mobile, with a live camera feed in Drive Mode
- **Automatic sign detection** — GPS + geohash proximity index triggers burst photo capture (3 frames, 2s apart) when within 100m of a known sign, with no user interaction required
- **AI vision analysis pipeline** — each uploaded frame triggers a Lambda that calls Claude claude-sonnet-4-6 via Bedrock; Claude returns a verdict (CLEAR / UNCLEAR / OBSCURED / MISSING) and natural-language reasoning
- **Review map** — Leaflet map showing all flagged signs from the session, grouped by sign so multiple captures collapse into one pin; tapping a pin shows an image gallery with per-frame Claude reasoning and a one-tap city reporting link
- **Sign index** — public road sign data for Seattle, King County, and Chicago fetched from city open data APIs (ArcGIS + Socrata), merged and cached in IndexedDB on the device
- **City discovery agent** — a Claude-powered script (`discover_city.py`) that can autonomously onboard a new city by searching the web for its open data portal, inspecting the API schema, and generating a validated config

## What Was Not Built

- User authentication — session IDs are random and stored in localStorage; no login
- Pre-signed S3 URLs — frames are uploaded directly from the browser with a public bucket write policy (the proper fix for a production app)
- Push notifications when analysis completes
- Historical data across sessions / multi-user aggregation
- Native mobile app — it's a PWA, so camera quality depends on browser support

## AWS Services Used

| Service | Role |
|---|---|
| **S3** | Stores uploaded camera frames and the sign index (`signs-latest.json`) |
| **Lambda** | `analyze-frame` — triggered by S3 uploads, calls Bedrock, writes flags to DynamoDB. `get-flags` — REST endpoint that returns session flags to the review screen |
| **Bedrock** | Claude claude-sonnet-4-6 (cross-region inference) — vision analysis of each captured frame |
| **API Gateway** | Exposes `GET /flags` for the review screen |
| **DynamoDB** | Stores flagged sign records (location, verdict, reasoning, session) |
| **Amplify** | Hosts the React PWA with HTTPS and global CDN |
| **SAM / CloudFormation** | Infrastructure as code — defines and deploys all of the above |

**Docker** was used locally during `sam build --use-container` to compile Python Lambda dependencies inside an Amazon Linux container matching the Lambda runtime.

## Security Considerations

Known limitations accepted for the scope of a hackathon:

- **Unauthenticated S3 writes** — the frames bucket allows public `PutObject` on `frames/*`. The bucket name is visible in the compiled JS bundle, so anyone could upload to it and trigger Bedrock analysis calls. The production fix is pre-signed upload URLs generated by a backend endpoint.
- **No API authentication** — the `GET /flags` endpoint is public; session IDs provide obscurity but not real auth.
- **Lambda concurrency cap** — `ReservedConcurrentExecutions: 10` on the analyze-frame function limits the blast radius of any abuse to 10 concurrent Bedrock calls.
- **Image privacy** — street-level frames (which may incidentally capture faces or license plates) are stored in S3 with public read access. A production version would restrict access and implement a retention policy.
